ARCHITECTURE

Événementielle.
Stateless. Modulaire.

Sept périphériques Z-Wave. Un middleware IoT. Trois modules PHP. Une base de données. L'architecture est simple par conception.

VUE D'ENSEMBLE

Quatre couches, un flux.

Chaque couche a une responsabilité unique. Les données circulent dans un seul sens : de l'espace physique vers la base de données. Le feedback circule en retour : de la base vers l'espace physique.

CYCLE DE VIE D'UN ÉVÉNEMENT

Du capteur à la base de données.

PHYSIQUE
Le joueur réalise une action physique

Appuie sur un bouton, ouvre une porte, traverse une zone.

Z-WAVE
Le périphérique Z-Wave transmet le changement d'état

Le périphérique Z-Wave détecte le changement d'état et le transmet au contrôleur Z-Stick USB.

DOMOTICZ
Domoticz reçoit la trame Z-Wave

Domoticz met à jour l'état du périphérique et déclenche le script dzVents correspondant.

WEBHOOK
dzVents envoie le webhook POST

dzVents appelle openURL() POST vers /api/handle_event.php avec token + idx + nvalue + svalue.

NORMALISATION
EventManager normalise le payload

EventManager::fromWebhook() résout le capteur par domoticz_idx, mappe nvalue vers code_evenement (DOOR_OPEN, BUTTON_PRESS...).

VALIDATION
GameEngine valide contre le scénario

GameEngine::process() charge la session active, charge etape_attend, compare l'événement à l'attendu (id_capteur + id_type_evenement).

TRANSITION
L'état de session évolue

En cas de succès : score mis à jour, étape suivante chargée, actions on_success exécutées. En cas d'échec : nb_erreurs incrémenté, actions on_failure exécutées.

PERSISTANCE
Tous les enregistrements écrits en base

Enregistrement evenement_session inséré. statut_session mis à jour. Enregistrements action_executee insérés.

MODULES

Trois modules. Trois responsabilités.

EventManager
  • · Reçoit le payload brut Domoticz (idx, nvalue, svalue)
  • · Résout le capteur par domoticz_idx (table capteur)
  • · Mappe type_capteur + nvalue vers code_evenement
  • · Retrouve l'enregistrement evenement_type
  • · Retourne : ['capteur', 'code_evenement', 'evenement_type', 'raw']
core/EventManager.php
GameEngine
  • · Charge la session active (statut_session = 'en_cours')
  • · Charge etape et etape_attend pour l'étape courante
  • · Appelle matchesAttend() pour comparer événement vs attendu
  • · Appelle onSucces() ou onEchec()
  • · Gère le cycle de vie de la session : démarrage, avancement, victoire
core/GameEngine.php
ActionManager
  • · Interroge etape_declenche JOIN actionneur JOIN action_type
  • · Dispatche par code_action : LCD_MESSAGE, PLUG_ON...
  • · Envoie GET HTTP au service Flask LCD
  • · Appelle DomoticzClient::turnOn/Off pour prises
core/ActionManager.php
DÉCISIONS DE CONCEPTION

Chaque choix a une raison.

Backend stateless

PHP ne conserve aucun état de session. Chaque requête lit depuis MariaDB. Le système est résistant aux pannes, redémarrable, et horizontalement scalable.

Webhook plutôt que polling

Domoticz pousse les événements vers PHP via webhook HTTP (dzVents openURL). Aucun processus PHP ne scrute Domoticz. Aucune latence supplémentaire.

Pas de boucle actionneur

Les actionneurs (prise, LCD) sont exclus du listener dzVents. Seuls les capteurs déclenchent des webhooks. Cela évite les boucles d'événements.

LCD comme microservice

L'écran LCD (PiFace, bus SPI) est contrôlé via un service Python Flask. Cela évite les problèmes de permissions SPI de www-data sur Raspberry Pi.
