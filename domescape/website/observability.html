In an event-driven physical system, visibility is critical. DomEscape logs every sensor event, every state transition, every action — enabling debugging, replay and level design analysis.

What is logged

From raw payload to game outcome.

⬤
Raw Domoticz Payload
Every incoming webhook stores the original idx, nvalue and svalue before any processing — enabling full replay from the source.
◈
Normalized Event Code
The mapped event code (DOOR_OPEN, BUTTON_PRESS...) is persisted alongside the raw data, showing exactly what the engine interpreted.
◎
Validation Result
is_expected (boolean) records whether the event matched what the engine expected — distinguishing valid actions from mistakes.
⧈
Session State Snapshots
game_sessions stores current_puzzle_id, score, mistake_count and status after every transition — a full timeline of session state.
⊡
Puzzle Context
Each event is linked to related_puzzle_id — making it trivial to query all events that occurred during a specific enigma.
◧
Precise Timestamps
event_time at second precision. Cross-referencing timestamps across session_events reveals exact durations per puzzle and total session time.

Debugging physical systems

When a sensor doesn't trigger as expected, the raw payload log reveals exactly what Domoticz sent — idx, nvalue, svalue. This eliminates ambiguity in diagnosing Z-Wave or Domoticz configuration issues.

The is_expected flag makes it immediate to query all events that were wrong actions — without needing to replay the session mentally.

Level design analytics

-- Which puzzle causes the most failures?
SELECT p.title, COUNT(*) as failures
FROM session_events se
JOIN puzzles p ON se.related_puzzle_id = p.id_puzzle
WHERE se.is_expected = 0
GROUP BY p.id_puzzle
ORDER BY failures DESC;
